June 1st 2014

Public Data for Public Lands

Written by

Alan Williams, PLATS Facilitator

Americans have inherited a vast, spectacular landscape. Through the tireless efforts of citizens and governments across generations, today we share access to over 785 million acres of public land. It's an incredible bounty we are charged with protecting for future generations.

In the 20th century, the conservation movement bloomed across all levels of government. Passionate citizens used any instrument available to protect the lands they cared for. As a result, thousands of organizations serve as the stewards of this endowment, ranging from Federal agencies, to state subdivisions, to cities, special districts, and non-profits.

Today, Code for America is pleased to announce a data collaboration with stewards across the country. We are developing a common format for public data about public lands--a commons for the 21st century.

The "Public Land and Trail Specification" aims to empower stewards to share data between one another and the visiting public. By publishing data in a standard set of tables and attributes, park agencies and other organizations can gain access to an evolving ecosystem of web tools, mobile apps, and data management platforms.

Over the next six weeks, Code for America is facilitating a public Request for Comment period. We are soliciting insights and constructive critique from domain experts and software developers alike. The comment period will culminate in an official version 1.0 of the specification.

Public data about public lands has the potential to improve coordination among stewards. Used wisely, public data can improve citizen's ability to safetly visit and experience our nation's parks and trails. In turn, increased visitation has the potential to advance economic development and public health goals.

Code for America is here to help governments be effective. If you are interested in publishing data compliant with PLATS, please get in touch. If you aren't sure whether PLATS is right for you, we are here to answer your questions. If you are a technologist looking to build products and services using PLATS data, we'd love to talk.

trailsegments.geojson
Geometry Linestring or MultiLineString Yes The geometry of the segment. n/a
internalId integer Yes The ID for this segment, which can be referenced. n/a
stewardId integer yes The ID of the steward responsibile for this trail segment, as defined in segments.json n/a
name string no Some segments have proper names of their own. This allows data providers to name segments, which are in turn accepted in OpenStreetMap n/a
highway string yes "highway=path" is an OpenStreetMap key value pair that indicates rights of way appropriate for non-motorized travel. "path"
motorVehicles string no Indicates whether a given trail segment is appropriate for motorized vehicles, including but not limited to OHV, ATVs, and snowmobiles. This field is derived from OpenStreetMap conventions. "yes", "no", "permissive", "designated"
foot string no Indicates whether walking and hiking is an appropriate use of this trail segment. This field is derived from OpenStreetMap conventions. "yes", "no", "permissive", "designated"
bicycle string no Indicates whether general bicycling is an appropriate use of this trail segment. This field is derived from OpenStreetMap conventions. "yes", "no", "permissive", "designated"
horse string no Indicates whether horseback riding or equestrian travel is an appropriate use of this trail segment. This field is derived from OpenStreetMap conventions. "yes", "no", "permissive", "designated"
ski string no Indicates whether over-land or cross-country skiing is an appropriate use of this trail segment. This field is derived from OpenStreetMap conventions. "yes", "no", "permissive", "designated"
wheelchair string no Indicates whether this trail segment is considered wheelchair accessible. This field is derived from OpenStreetMap conventions. Other attributes that describe accessibility can be included in the 'osmTags" field. "yes", "no", "permissive", "designated"
osmTags array of strings no This field is designed to incorporate a fungible number of key-value pairs that describe segments within OpenStreetMap. Anticipated use of this field includes the incorporation of keys such as "width", "surface" and "mtnbike".

name string Yes The proper name of the trail, as advertised to visitors and public communications. n/a
segmentIds array of strings Yes An array of IDs, which allows applications to construct "Named Trails" from a collection of one or more trail segment geometries. n/a
id integer Yes A unique id for the named trail. Currently, this ID is used to identify which Named Trails each respective Steward is responsible for maintaing and/or handling inquiries regarding. n/a
descriptn string Yes This is the primary field for narrative descriptions of a Named Trail, and the experience or features a visitor might expect to encounter. n/a
dsgnations array of string No This field allows for data contributors to indicate whether a given trail has been bestowed with particular nationl, state or local designations or distinctions. n/a
partOf array of string No This field allows for data contributors to indicate a relationship between Named Trails that share constiuent segments. The anticipated use case is to indicate that a trail serves a leg in a larger regional or national trail. n/a

trailheads.geojson
geom string Yes The point geometry for the trailhead. n/a
name array of strings Yes The public name of the trailhead, as advertised to visitors. n/a
trailIds integer Yes A collection of IDs for Named Trails (match to "id" in namedtrails.json), each representing a trail that can be accessed via this trailhead. n/a
stewardIds string Yes A collection of IDs for Stewards (match to "id" stewards.json), representing each agency or organization responsive to visitor concerns or inquiries about this trailhead. n/a
parkName array of string No A collection of park names that visitors should associate with each trailhead. These park names currently exist independently from other aspects of the specification. n/a
address array of string No The street address, if one exists, that is associated with this trialhead. n/a
parking array of string No An indication as to whether visitors can expect to find automobile park at or immediately adjacent to this trailhead. n/a
drinkwater array of string No An indication as to whether visitors can expect to find drinking water available at or immediately adjacent to this trailhead. n/a
